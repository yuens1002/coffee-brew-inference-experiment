# Plans & ACs

Feature plans and acceptance criteria docs live here, one directory per feature.

## Convention

```
docs/plans/
  {feature}-plan.md      ← deliverables list, branch name, commit schedule
  {feature}-ACs.md       ← AC table (UI / Functional / Test / Regression)
  {feature}-review.md    ← /review report (generated post-verification)
```

- Plan and ACs are created together, before implementation starts.
- AC doc is never created after implementation — that defeats the purpose.
- Review reports are generated by `/review` after the verify sub-agent's pass.

## AC table columns

| Column | Who fills it | When |
|--------|-------------|------|
| AC, What, How, Pass | PM / engineer | At plan time |
| Agent | Verify sub-agent | During Phase 3 verify |
| QC | QC reviewer | During Phase 4 |
| Reviewer | Human | During Phase 5 human review |

## Verification methods

| Method | Use for |
|--------|---------|
| `Code review: path/to/file.ts` | Structure, typing, logic correctness |
| `Test run: npm test` | Automated test assertions |
| `E2E: curl ...` | Live endpoint shape verification |
| `Static: screenshot` | UI layout (n/a for this project currently) |

## Branch naming

`feat/{short-description}` — derived from the plan's primary deliverable.  
`fix/{scope}-{short-description}` — for bug fix branches.
